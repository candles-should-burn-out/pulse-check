# ADR-0003: Наборы статусов

Status: Accepted

## Context

Pulse Check использует статусы для учета сущностей и статистики. Статусу нужен неизменяемый технический идентификатор, чтобы агрегаты не ломались при изменении имени или цветов.

Модель участников требует общий справочник статусов: участник должен видеть тот же набор, что и пригласивший пользователь, но не должен менять его. Это решение уточняет ADR-0001: имена и цвета статусов являются осознанным серверным исключением из local-first границы, потому что они нужны для совместной статистики.

## Decision

Ввести server-managed status set. Backend является источником истины для наборов статусов, самих статусов и связи пользователя с набором.

Ключевые правила:

- каждый аккаунт имеет один status set;
- при первом обращении пользователя backend создает собственный `status_set` и membership владельца;
- `status_id` является неизменяемым UUID, который генерирует backend;
- владелец может создавать, редактировать и удалять статусы своего набора;
- участник использует status set пригласившего пользователя в режиме только для чтения;
- роль вычисляется из `status_set_memberships`: владелец, если `user_id` равен `owner_user_id`, иначе участник;
- backend хранит итоговые цвета, а не идентификатор frontend-палитры;
- список статусов возвращается целиком без пагинации.

## User-flow

### New Flow: Первое получение набора статусов

Потребитель: user

Authenticated user открывает приложение или страницу статусов. Если membership еще нет, backend создает новый `status_set`, связывает его с пользователем как владельцем и возвращает пустой список статусов.

### New Flow: Управление статусами

Потребитель: user

Владелец открывает `/app/statuses`, создает, редактирует и удаляет статусы. Frontend отправляет изменения в backend и показывает значения, сохраненные сервером.

### Exist Flow: Просмотр унаследованных статусов

Потребитель: participant

Участник открывает `/app/statuses` и видит status set пригласившего пользователя в read-only режиме.

### Exist Flow: Просмотр статистики по статусам

Потребитель: user

Пользователь открывает `/app/statistics/statuses` и видит статусы из набора как основу для агрегированной статистики.

## Configuration

- `STATUS_SET_MAX_STATUSES` - максимальное количество статусов в одном наборе, по умолчанию `50`.
- Frontend-палитры статусов являются статической frontend-конфигурацией и не требуют backend API.

## Database

Backend хранит наборы статусов в PostgreSQL.

### Таблица `status_sets`

Хранит наборы статусов и владельцев.

| Поле            | Тип           | Constraint | Ограничения                                                 |
|-----------------|---------------|------------|-------------------------------------------------------------|
| `id`            | `UUID`        | not null   | Первичный ключ, генерируется backend-ом                     |
| `owner_user_id` | `TEXT`        | not null   | Уникальный идентификатор владельца из контекста авторизации |
| `created_at`    | `TIMESTAMPTZ` | not null   | Дата создания записи                                        |
| `updated_at`    | `TIMESTAMPTZ` | not null   | Дата последнего изменения записи                            |

Дополнительные заметки:

- `owner_user_id` - один владелец не может иметь больше одного собственного status set.

### Таблица `status_set_memberships`

Хранит status set пользователя.

| Поле            | Тип           | Constraint | Ограничения                                                                                               |
|-----------------|---------------|------------|-----------------------------------------------------------------------------------------------------------|
| `user_id`       | `TEXT`        | not null   | Первичный ключ; один status set на аккаунт                                                                |
| `status_set_id` | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `status_set_memberships_status_set_id_idx` |
| `owner_user_id` | `TEXT`        | not null   | Идентификатор владельца status set                                                                        |
| `created_at`    | `TIMESTAMPTZ` | not null   | Дата создания связи                                                                                       |

Дополнительные заметки:

- `user_id` - если пользователь входит без связи, backend создает для него собственный `status_set` и membership владельца.
- `owner_user_id` - используется для вычисления роли: `status_owner` или `participant`.

### Таблица `statuses`

Хранит определения статусов внутри набора.

| Поле               | Тип           | Constraint | Ограничения                                                                                                 |
|--------------------|---------------|------------|-------------------------------------------------------------------------------------------------------------|
| `id`               | `UUID`        | not null   | Первичный ключ, генерируется backend-ом и не меняется                                                       |
| `status_set_id`    | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `statuses_status_set_id_idx`                 |
| `name`             | `TEXT`        | not null   | Не длиннее 40 символов; check constraint `char_length(name) <= 40`; constraint `statuses_name_length_check` |
| `border_color`     | `CHAR(7)`     | not null   | HEX-цвет в формате `#RRGGBB`; check constraint по regex `^#[0-9A-Fa-f]{6}$`                                 |
| `background_color` | `CHAR(7)`     | not null   | HEX-цвет в формате `#RRGGBB`; check constraint по regex `^#[0-9A-Fa-f]{6}$`                                 |
| `text_color`       | `CHAR(7)`     | not null   | HEX-цвет в формате `#RRGGBB`; check constraint по regex `^#[0-9A-Fa-f]{6}$`                                 |
| `created_at`       | `TIMESTAMPTZ` | not null   | Дата создания записи                                                                                        |
| `updated_at`       | `TIMESTAMPTZ` | not null   | Дата последнего изменения записи                                                                            |

Дополнительные заметки:

- `name` - ограничение 40 символов проверяется на frontend и backend.

## API

Все ручки статусов требуют authenticated user из Bearer JWT, если OIDC включен.

### GET: /status-set

Тело запроса: отсутствует

Код ответа: `200 OK`

Ответ:

```json
{
  "id": "7d2a6f2b-3d4e-4f6a-9a8b-111111111111",
  "owner_user_id": "owner-user-id",
  "role": "status_owner",
  "statuses": [
    {
      "id": "f3a5d6e7-8b9c-4d10-9e11-222222222222",
      "name": "Активен",
      "border_color": "#2F6BFF",
      "background_color": "#EAF0FF",
      "text_color": "#102A6B",
      "created_at": "2026-05-15T10:00:00Z",
      "updated_at": "2026-05-15T10:00:00Z"
    }
  ]
}
```

Права доступа: владелец получает `role: "status_owner"`, участник получает `role: "participant"`.

### GET: /statuses

Тело запроса:

```json

  "name": "Активен",
  "border_color": "#2F6BFF",

  "text_color": "#102A6B"

```

Код ответа: `201 Created`

Ответ:

```json
{
  "id": "f3a5d6e7-8b9c-4d10-9e11-222222222222",
  "name": "Активен",
  "border_color": "#2F6BFF",
  "background_color": "#EAF0FF",
  "text_color": "#102A6B",
  "created_at": "2026-05-15T10:00:00Z",
  "updated_at": "2026-05-15T10:00:00Z"
}
```

Права доступа: только владелец status set.

Ошибки:

Код ответа: `400 Bad Request`

```json
{
  "error": "invalid_status"
}
```

Код ответа: `403 Forbidden`

```json
{
  "error": "status_set_read_only"
}
```

Код ответа: `409 Conflict`

```json
{
  "error": "status_limit_reached"
}
```

Тело запроса: отсутствует

## Страницы

### /app/statuses

Назначение: управление набором статусов текущего пользователя.

Доступ: authenticated user. Владелец видит режим управления, участник - read-only режим.

Основные элементы:

- список статусов;
- preview статуса с выбранными цветами;
- dialog создания и редактирования для владельца;
- поле имени, color inputs и frontend-палитры;
- read-only уведомление для участника.

Действия:

- создать статус: frontend отправляет `POST /statuses` и добавляет ответ backend в список;
- изменить статус: frontend отправляет `PATCH /statuses/{status_id}` и сохраняет неизменный `id`;
- удалить статус: frontend подтверждает действие и отправляет `DELETE /statuses/{status_id}`;
- выбрать frontend-палитру или цвета вручную: frontend отправляет итоговые HEX-цвета.

Состояния:

- загрузка status set;
- пустой список;
- ошибка загрузки или сохранения;
- ошибка валидации имени длиннее 40 символов;
- ошибка `status_limit_reached`;
- read-only режим участника;
- успешное создание, изменение и удаление.

### /app/statistics/statuses

Назначение: read-only экран статусов как основы статистики по набору.

Доступ: authenticated user.

Основные элементы:

- список статусов из status set;
- роль пользователя относительно набора;
- пустое состояние, если статусы не настроены.

Действия:

- просмотреть статусы, используемые для статистики.

Состояния:

- загрузка статусов;
- пустой список;
- ошибка загрузки;
- read-only режим.

## Ограничения

- Максимальное количество статусов в одном наборе по умолчанию равно 50.
- Имя статуса ограничено 40 символами.
- `border_color`, `background_color` и `text_color` должны быть HEX-цветами в формате `#RRGGBB`.
- Пагинации для статусов нет.
- Участник не может создавать, редактировать или удалять статусы.
- Пользователь не должен использовать в названиях статусов персональные, чувствительные, оскорбительные, дискриминационные или специальные категории персональных данных.

## Consequences

- `status_id` становится стабильным ключом для сущностей и агрегатов.
- Владелец и участники видят один и тот же набор статусов, поэтому статистика остается сопоставимой.
- Backend хранит пользовательские имена и цвета статусов, поэтому они должны соответствовать data minimization ограничениям.
- API статусов возвращает полный справочник; масштабирование ограничено `STATUS_SET_MAX_STATUSES`.
- PostgreSQL становится продуктовым хранилищем определений статусов, отдельно от Keycloak.
- Удаление используемого статуса временно может оставлять сущности без валидного статуса до отдельной доработки.

## Проработка

- Спроектировать сценарий удаления или переназначения статуса, который уже используется сущностями.
- Расширить `/app/statistics/statuses` до полноценной статистики по статусам и участникам без раскрытия конкретных сущностей.
