# Правила составления ADR

Этот файл фиксирует структуру и требования к новым Architecture Decision Records в Pulse Check. При подготовке следующих ADR нужно следовать этим правилам.

## Название и Метаданные

ADR должен начинаться с заголовка:

```md
# ADR-NNNN: Краткое название решения
```

Ниже обязательно указывается:

```md
Status: In Progress | Accepted | Suspended
```

Номер ADR должен быть следующим свободным номером в `docs/adr`. Название должно описывать принятое архитектурное решение, а не задачу реализации.

## Context

Раздел описывает проблему, ограничения и причины, из-за которых нужно принять решение.

В этом разделе нужно:

- описать текущую ситуацию;
- указать бизнесовые, продуктовые, технические или privacy-факторы;
- назвать связанные ADR, если решение уточняет или меняет их границы.

## Decision

Раздел описывает принятое решение и его границы.

В этом разделе нужно:

- формулировать решение явно;
- перечислять ключевые правила поведения системы;
- фиксировать, что является источником истины;
- указывать важные значения по умолчанию, лимиты и конфигурационные параметры;
- не смешивать детали базы данных и API с основным текстом, если для них нужны отдельные подразделы ниже.

## Данные и База Данных

Все, что связано с созданием или изменением таблиц в базе данных, должно быть вынесено в этот раздел.

Если реализация подразумевает создание или изменение таблиц, для каждой таблицы нужно добавить Markdown-таблицу со строгими колонками:

| Поле | Тип    | Constraint | Ограничения                             |
|------|--------|------------|-----------------------------------------|
| `id` | `UUID` | not null   | Первичный ключ, генерируется backend-ом |

Требования к описанию таблиц:

- Constraint должен содержать только `null` или `not null`
- каждая таблица описывается отдельно;
- все поля таблицы должны быть перечислены;
- первичные ключи, внешние ключи, уникальность, default-значения, check-ограничения и индексы описываются в колонке `Ограничения`;

Если по отдельным полям нужны дополнительные пояснения, их нужно добавить ниже таблицы списком:

```md
Дополнительные заметки:

- `field_name` - комментарий.
- `other_field` - комментарий.
```

## API

Все, что связано с backend API, должно быть вынесено в этот раздел.

Каждая ручка описывается отдельно в формате:

~~~md
### POST: /path

Тело запроса:

```json
{
  "field": "value"
}
```

Код ответа: `200 OK`

Ответ:

```json
{
  "id": "uuid",
  "field": "value"
}
```
~~~

Требования к описанию API:

- указывать HTTP-метод и путь в заголовке ручки в формате `### METHOD: /path`;
- если тела запроса нет, писать `Тело запроса: отсутствует`;
- если тела ответа нет, писать `Ответ: отсутствует`;
- для ошибок указывать код ответа и пример JSON, если контракт ошибки важен для клиента;
- отдельно фиксировать права доступа, если ручка отличается для владельца, участника, администратора или анонимного пользователя;
- если API не затрагивается, раздел должен содержать короткую строку `Не затрагивается`.

## Страницы

Все, что связано с frontend-страницами, которые нужно создать или изменить, должно быть вынесено в этот раздел.

Каждая страница описывается отдельно в формате:

```md
### /app/example

Назначение: краткое описание задачи страницы.

Доступ: кто может открыть страницу и какие роли видят разные режимы.

Основные элементы:

- список, форма, таблица, фильтры, панель действий или другие ключевые части интерфейса;
- важные поля, которые пользователь видит или редактирует.

Действия:

- создать, изменить, удалить, сохранить, отменить, экспортировать или другое пользовательское действие;
- для каждого действия указать ожидаемый результат.

Состояния:

- загрузка;
- пустое состояние;
- ошибка;
- read-only режим, если применимо;
- успешное сохранение или другое важное состояние.
```

Требования к описанию страниц:

- указывать точный route страницы;
- описывать назначение страницы через пользовательскую задачу, а не через техническую реализацию;
- отдельно фиксировать различия поведения для ролей и режимов доступа;
- перечислять обязательные элементы интерфейса и пользовательские действия;
- указывать состояния загрузки, ошибки, пустого результата и режима только для чтения, если они применимы;
- если страницы не затрагиваются, раздел должен содержать короткую строку `Не затрагивается`.

## Ограничения

Раздел фиксирует ограничения решения.

В этом разделе нужно перечислять:

- лимиты и настраиваемые параметры;
- ограничения безопасности и приватности;
- ограничения валидации на frontend и backend;
- осознанно не поддержанные сценарии;
- зависимости от внешних сервисов или инфраструктуры.

## Consequences

Раздел описывает последствия принятого решения.

В этом разделе нужно:

- перечислить положительные последствия;
- перечислить компромиссы и риски;
- указать изменения в ответственности frontend, backend, инфраструктуры или пользователя;
- отдельно назвать влияние на privacy boundary, если решение касается пользовательских данных.

## Проработка

Раздел содержит вопросы и задачи, которые не решены текущим ADR.

В этом разделе нужно фиксировать:

- сценарии, требующие отдельного дизайна;
- технические вопросы, которые нужно проверить;
- будущие ADR или доработки, если решение сознательно оставляет часть поведения за рамками;
- вопросы миграции, обратной совместимости или эксплуатации, если они не решены полностью.

Если открытых вопросов нет, раздел должен содержать строку `Открытых вопросов нет`.

