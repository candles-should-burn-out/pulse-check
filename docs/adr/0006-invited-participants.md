# ADR-0006: Приглашенные участники и агрегированная статистика

Status: Accepted

## Context

Pulse Check должен поддержать сценарий, в котором один пользователь приглашает других пользователей в общий рабочий контекст. Приглашенный пользователь использует статусы пригласившего, не может менять определения статусов и отправляет на сервер только технические агрегаты для общей статистики.

Это решение уточняет ADR-0001, ADR-0003 и ADR-0005:

- user-owned data остаются локальными;
- набор статусов остается server-managed и read-only для всех участников, кроме владельца набора;
- локальные имена участников не отправляются на backend.

## Decision

Ввести модель приглашенных участников. Пользователь может создать ссылку приглашения или QR-код, передать ее другому человеку, а новый пользователь после принятия приглашения получает эффективный набор статусов пригласившего пользователя.

Ключевые правила:

- продуктовый термин для UI - "участник";
- "владелец" - пользователь, которому принадлежит `status_set`;
- "пригласивший" - пользователь, который пригласил другого человека и поделился с ним используемым `status_set`;
- "участник" - пользователь, принявший приглашение и использующий `status_set` пригласившего;
- каждый user может иметь только один `status_set`;
- участник наследует `status_set` пригласившего пользователя;
- участник может просматривать статусы, но не может создавать, редактировать или удалять их;
- участник может приглашать других пользователей в тот же `status_set`;
- QR-код является frontend-представлением той же ссылки приглашения и не требует отдельной backend-сущности;
- ссылка приглашения должна передавать token во frontend fragment, например `/join#token=...`, чтобы token не уходил в backend path, query string и access logs;
- ссылка приглашения создается backend-ом как одноразовый или ограниченно многоразовый token;
- backend хранит только hash token-а, а plaintext token возвращает только при создании приглашения;
- первый вход приглашенного пользователя проходит через Keycloak invite-scoped registration/login flow;
- Pulse Check backend не хранит пароль и не является источником истины для учетных записей;
- после успешного login frontend вызывает backend для принятия приглашения текущим authenticated user;
- backend создает связь участника с `status_set` пригласившего;
- backend хранит агрегированную статистику по `status_id` и `user_id`;
- владелец видит агрегированную статистику всех участников своего `status_set`;
- участник видит собственную агрегированную статистику и агрегированную статистику участников, которых он пригласил сам или через свою ветку приглашений;
- список участников возвращается backend-ом только с техническими идентификаторами и связями приглашений;
- пользовательские имена участников хранятся только локально на frontend и ключуются по `status_set_id` и `user_id`;
- удаление участника удаляет его связь, ветку приглашенных им участников и их агрегированную статистику из этого `status_set`;
- удаление участника не удаляет его учетную запись в Keycloak.

Источники истины:

- Keycloak является источником истины для учетных записей, login, password, блокировок и password reset;
- backend является источником истины для приглашений, membership graph и агрегированных счетчиков;
- локальное хранилище frontend является источником истины для локальных имен участников;
- локальное хранилище пользователя остается источником истины для конкретных сущностей.

Основные параметры конфигурации:

- `INVITATION_TOKEN_TTL_HOURS` - срок жизни ссылки приглашения, по умолчанию 168 часов;
- `INVITATION_DEFAULT_MAX_USES` - количество использований одной ссылки по умолчанию, по умолчанию 1;
- `INVITATION_MAX_USES_LIMIT` - максимальное количество использований одной ссылки, по умолчанию 25;
- `PARTICIPANT_TREE_MAX_DEPTH` - максимальная глубина дерева приглашений, по умолчанию 5;
- `PARTICIPANT_MAX_PER_STATUS_SET` - максимальное количество участников в одном `status_set`, по умолчанию 100.

## Данные и База Данных

Backend хранит приглашения, дерево участников и агрегированные счетчики в PostgreSQL.

Таблица `status_set_memberships` изменяется и после этого содержит:

| Поле                     | Тип           | Constraint | Ограничения                                                                                               |
|--------------------------|---------------|------------|-----------------------------------------------------------------------------------------------------------|
| `user_id`                | `TEXT`        | not null   | Первичный ключ; один эффективный набор статусов на аккаунт                                                |
| `status_set_id`          | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `status_set_memberships_status_set_id_idx` |
| `owner_user_id`          | `TEXT`        | not null   | Идентификатор владельца эффективного набора статусов                                                      |
| `invited_by_user_id`     | `TEXT`        | null       | Идентификатор пригласившего пользователя; `null` для владельца                                            |
| `accepted_invitation_id` | `UUID`        | null       | Внешний ключ на `participant_invitations.id`; `null` для владельца                                        |
| `created_at`             | `TIMESTAMPTZ` | not null   | Дата создания связи, выставляется backend-ом                                                              |

Дополнительные заметки:

- `status_set_memberships.user_id` - если удаленный участник снова входит в продукт без нового приглашения, backend должен создать для него собственный `status_set` как для нового владельца.
- `status_set_memberships.invited_by_user_id` - формирует дерево приглашений внутри одного `status_set`; значение должно ссылаться на пользователя из того же `status_set`.
- `status_set_memberships.accepted_invitation_id` - нужно для аудита принятия приглашения и диагностики конфликтов.
- `status_set_memberships.status_set_id`, `status_set_memberships.user_id` - уникальный индекс `status_set_memberships_status_set_user_idx` нужен для внешних ключей агрегированной статистики.

Таблица `participant_invitations`:

| Поле                 | Тип           | Constraint | Ограничения                                                                                                  |
|----------------------|---------------|------------|--------------------------------------------------------------------------------------------------------------|
| `id`                 | `UUID`        | not null   | Первичный ключ, генерируется backend-ом                                                                      |
| `status_set_id`      | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `participant_invitations_status_set_id_idx`   |
| `owner_user_id`      | `TEXT`        | not null   | Идентификатор владельца эффективного набора статусов                                                         |
| `invited_by_user_id` | `TEXT`        | not null   | Идентификатор пользователя, создавшего приглашение                                                           |
| `token_hash`         | `TEXT`        | not null   | Уникальный hash token-а; plaintext token не хранится                                                         |
| `max_uses`           | `INTEGER`     | not null   | От 1 до `INVITATION_MAX_USES_LIMIT`; по умолчанию `INVITATION_DEFAULT_MAX_USES`                              |
| `accepted_uses`      | `INTEGER`     | not null   | Количество успешных принятий приглашения; default 0                                                          |
| `expires_at`         | `TIMESTAMPTZ` | not null   | Дата истечения срока действия                                                                                |
| `revoked_at`         | `TIMESTAMPTZ` | null       | Дата ручного отзыва приглашения                                                                              |
| `created_at`         | `TIMESTAMPTZ` | not null   | Дата создания записи, выставляется backend-ом                                                                |

Дополнительные заметки:

- `participant_invitations.token_hash` - вычисляется через cryptographic hash от случайного token-а с достаточной entropy.
- `participant_invitations.max_uses` и `accepted_uses` - позволяют использовать один QR-код для нескольких участников в ограниченном сценарии.
- `participant_invitations.revoked_at` - отозванное приглашение нельзя принять, даже если срок действия не истек.

Таблица `entity_status_counters`:

| Поле            | Тип           | Constraint | Ограничения                                                                                        |
|-----------------|---------------|------------|----------------------------------------------------------------------------------------------------|
| `status_set_id` | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`                                             |
| `user_id`       | `TEXT`        | not null   | Идентификатор пользователя, которому принадлежат локальные сущности                                |
| `status_id`     | `UUID`        | not null   | Внешний ключ на `statuses.id` с `ON DELETE CASCADE`; индекс `entity_status_counters_status_id_idx` |
| `count`         | `INTEGER`     | not null   | Количество локальных сущностей пользователя в этом статусе; не меньше 0                            |
| `updated_at`    | `TIMESTAMPTZ` | not null   | Дата последнего обновления агрегата, выставляется backend-ом                                       |

Ограничения таблицы:

- составной первичный ключ: `status_set_id`, `user_id`, `status_id`;
- внешний ключ: `status_set_id`, `user_id` ссылается на `status_set_memberships.status_set_id`, `status_set_memberships.user_id` с `ON DELETE CASCADE`.

Дополнительные заметки:

- `entity_status_counters` хранит только агрегаты и не содержит идентификаторы конкретных сущностей.
- При удалении участника backend удаляет его строки `entity_status_counters` вместе со строками удаляемой ветки участников.

## API

Все product API ручки, кроме проверки ссылки приглашения, требуют authenticated user из Bearer JWT, если OIDC включен.

### POST: /participant-invitations

Тело запроса:

```json
{
  "max_uses": 1,
  "expires_in_hours": 168
}
```

Код ответа: `201 Created`

Ответ:

```json
{
  "id": "7d2a6f2b-3d4e-4f6a-9a8b-111111111111",
  "invite_url": "https://pulse-check.example/join#token=plaintext-token",
  "token": "plaintext-token",
  "max_uses": 1,
  "accepted_uses": 0,
  "expires_at": "2026-05-23T10:00:00Z",
  "created_at": "2026-05-16T10:00:00Z"
}
```

Права доступа: доступно владельцу и участнику. Приглашение создается в эффективный `status_set` текущего пользователя.

Ошибки:

Код ответа: `400 Bad Request`

```json
{
  "error": "invalid_invitation"
}
```

Код ответа: `409 Conflict`

```json
{
  "error": "participant_limit_reached"
}
```

### POST: /participant-invitations/validate

Тело запроса:

```json
{
  "token": "plaintext-token"
}
```

Код ответа: `200 OK`

Ответ:

```json
{
  "valid": true,
  "expires_at": "2026-05-23T10:00:00Z"
}
```

Права доступа: доступно анонимному и authenticated user. Ответ не раскрывает `owner_user_id`, `invited_by_user_id`, имена пользователей или статистику.

Ошибки:

Код ответа: `404 Not Found`

```json
{
  "error": "invitation_not_found"
}
```

Код ответа: `410 Gone`

```json
{
  "error": "invitation_expired"
}
```

### POST: /participant-invitations/accept

Тело запроса:

```json
{
  "token": "plaintext-token"
}
```

Код ответа: `201 Created`

Ответ:


Права доступа: доступно authenticated user, который еще не имеет эффективного `status_set`.

```json

}
```

```json

```


Код ответа: `404 Not Found`

```json

  "error": "participant_not_found"

```

## Страницы

### /join

Назначение: принять приглашение по ссылке или QR-коду.

Доступ: анонимный пользователь и authenticated user. Анонимный пользователь сначала проходит invite-scoped registration/login flow в Keycloak, затем возвращается на страницу принятия приглашения. Token приглашения читается frontend-ом из URL fragment и не отправляется в backend path или query string.

Основные элементы:

- состояние проверки ссылки приглашения;
- действие входа или регистрации через Keycloak;
- подтверждение принятия приглашения;
- сообщение об истекшей или отозванной ссылке;
- сообщение о конфликте, если аккаунт уже привязан к другому `status_set`.

Действия:

- открыть ссылку приглашения - frontend читает token из URL fragment, сохраняет его в session storage и вызывает `POST /participant-invitations/validate`;
- войти или зарегистрироваться - пользователь проходит Keycloak flow и возвращается на `/join`;
- принять приглашение - frontend отправляет `POST /participant-invitations/accept`;
- перейти в приложение - после успешного принятия пользователь попадает в `/app`.

Состояния:

- загрузка проверки token-а;
- приглашение действительно;
- приглашение истекло, отозвано или исчерпало лимит использований;
- пользователь не authenticated;
- аккаунт уже имеет эффективный `status_set`;
- успешное принятие приглашения;
- ошибка принятия приглашения.

### /app/participants

Назначение: управлять приглашениями, локальными именами участников и удалением участников.

Доступ: authenticated user. Владелец видит всех участников своего `status_set`. Участник видит себя и участников из своей ветки приглашений.

Основные элементы:

- список участников, доступных текущему пользователю;
- локальное имя участника, хранящееся только в frontend storage;
- техническое состояние участника без отображения backend-owned персональных данных;
- кнопка создания ссылки приглашения;
- QR-код для созданной ссылки;
- список активных приглашений текущего пользователя;
- действие отзыва приглашения;
- действие удаления участника.

Действия:

- создать приглашение - frontend отправляет `POST /participant-invitations`, показывает ссылку и генерирует QR-код локально;
- скопировать ссылку - frontend копирует `invite_url` в clipboard;
- показать QR-код - frontend кодирует `invite_url` без отдельного backend API;
- отозвать приглашение - frontend отправляет `DELETE /participant-invitations/{invitation_id}`;
- задать локальное имя участнику - frontend сохраняет имя локально и не отправляет его на backend;
- удалить участника - frontend отправляет `DELETE /participants/{user_id}`, после успеха удаляет локальное имя и обновляет список;
- просмотреть статистику участника - пользователь переходит к `/app/statistics/statuses` с локальным фильтром по `user_id`.

Состояния:

- загрузка списка участников и приглашений;
- пустое состояние без участников;
- пустое состояние без активных приглашений;
- ошибка создания приглашения;
- ошибка отзыва приглашения;
- ошибка удаления участника;
- ошибка достижения лимита участников;
- успешное создание приглашения;
- успешное удаление участника;
- read-only отображение backend-owned данных без передачи локальных имен.

### /app/statistics/statuses

Назначение: просматривать агрегированную статистику по статусам, участникам и суммарно по доступной ветке приглашений.

Доступ: authenticated user. Владелец видит всех участников своего `status_set`. Участник видит себя и участников из своей ветки приглашений.

Основные элементы:

- агрегированная статистика по статусам текущего пользователя;
- агрегированная статистика по каждому доступному участнику;
- суммарная статистика владельца и доступных участников;
- локальные имена участников из frontend storage;
- fallback на технический label, если локальное имя не задано.

Действия:

- просмотреть статистику по каждому участнику;
- переключить режим отображения между разрезом по участникам и суммарным видом;
- отфильтровать статистику по участнику;
- обновить агрегаты после локального изменения сущностей текущего пользователя через `PUT /statistics/status-counters`.

Состояния:

- загрузка агрегатов;
- пустое состояние, если агрегатов еще нет;
- ошибка загрузки агрегатов;
- участник удален и его локальное имя больше не используется;
- read-only режим без действий изменения чужой статистики;
- отсутствие раскрытия конкретных сущностей, имен, notes, attributes и персональных маппингов.

## Ограничения

- Ссылка приглашения по умолчанию живет 168 часов.
- Ссылка приглашения по умолчанию одноразовая.
- Один QR-код может быть многоразовым только в пределах `INVITATION_MAX_USES_LIMIT`.
- Максимальная глубина дерева приглашений по умолчанию равна 5.
- Максимальное количество участников в одном `status_set` по умолчанию равно 100.
- Участник не может менять определения статусов.
- Участник не может принять приглашение, если его аккаунт уже имеет эффективный `status_set`.
- Backend не хранит локальные имена участников.
- Backend не хранит конкретные сущности, notes, attributes или персональные маппинги.
- Backend не должен логировать plaintext invitation token.
- Удаление участника из Pulse Check не удаляет учетную запись в Keycloak.
- Invite-scoped registration/login требует настройки Keycloak и не должен включать публичную self-registration без token-а приглашения.

## Consequences

- Пользователи могут безопасно расширять общий рабочий контекст через ссылки и QR-коды.
- Статистика владельца и участников остается сопоставимой, потому что все используют один `status_set`.
- Участники могут строить собственные ветки приглашений без выдачи прав на редактирование статусов.
- Backend получает новую ответственность за invitation lifecycle, membership graph и агрегированные счетчики.
- Frontend получает ответственность за локальные имена участников, QR-код, clipboard UX и отображение статистики в разрезах.
- Keycloak остается ответственным за учетные записи и пароль, поэтому Pulse Check не реализует хранение паролей.
- Privacy boundary сохраняется: сервер хранит технические связи и агрегаты, но не хранит пользовательские имена участников и конкретные сущности.
- Удаление участника необратимо удаляет его агрегированную статистику в данном `status_set`, но не затрагивает его локальные данные на устройстве.

## Проработка

- Спроектировать точный Keycloak invite-scoped registration/login flow: extension, theme, required actions или другой механизм, который разрешает регистрацию только по token-у приглашения.
- Решить, нужен ли UI для многоразовых приглашений в v1 или достаточно одноразовых ссылок.
- Спроектировать UX подтверждения удаления участника с отображением количества участников в удаляемой ветке.
- Определить, как frontend синхронизирует `PUT /statistics/status-counters`: при каждом изменении сущностей, debounce, manual sync или background sync.
- Спроектировать миграцию для существующих пользователей: создание self-membership и перенос текущих статусов в собственный `status_set`.
