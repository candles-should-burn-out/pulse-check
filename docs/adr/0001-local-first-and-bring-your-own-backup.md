# ADR-0001: Local-first and Bring Your Own Backup

Status: Accepted

## Context

Pulse Check работает с пользовательскими названиями, заметками, атрибутами и потенциально сведениями о других людях. Продукт должен минимизировать доступ сервера к этим данным и явно разделять ответственность: пользователь владеет своими данными, а сервер отвечает только за согласованные исключения.

Сервер нужен для аутентификации, координации и агрегированной статистики, но не должен быть основным хранилищем user-owned data. Это базовая privacy boundary для следующих решений, включая ADR-0003 и ADR-0004.

## Decision

Pulse Check строится как local-first приложение. Локальное хранилище пользователя является источником истины для user-owned data.

Backup следует модели Bring Your Own Backup:

- пользователь явно запускает export/import;
- backup archive шифруется до передачи во внешнее хранилище;
- пользователь самостоятельно хранит экспортированный backup;
- архив переносим между устройствами пользователя;
- импорт проверяет структуру и integrity до изменения локальных данных.

Сервер по умолчанию не хранит пользовательские заметки, атрибуты, произвольные поля и персональные маппинги. Любое исключение из этой границы должно быть описано отдельным ADR.

Планируемый backup archive:

- `manifest.json` - версия формата, версия приложения и metadata backup;
- `data.json` - сериализованные локальные данные пользователя;
- `attachments/` - опциональные вложения;
- `checksums.json` - контрольные суммы файлов архива.

Для ZIP используется `fflate`, для валидации - Zod, для checksums - Web Crypto API/SHA-256.

## User-flow

### New Flow: Экспорт backup

Потребитель: user

Пользователь запускает export, приложение собирает локальные данные, валидирует структуру архива, шифрует backup и отдает файл пользователю.

### New Flow: Импорт backup

Потребитель: user

Пользователь выбирает backup archive, приложение проверяет формат, checksums и совместимость версии, затем импортирует данные в локальное хранилище.

## Ограничения

- Browser storage persistence снижает риск потери данных, но не заменяет backup.
- Пользователь отвечает за хранение backup вне приложения.
- Backup должен быть доступен приложению только после явного действия пользователя.
- Shared readable data и серверные исключения из local-first границы требуют отдельного ADR.

## Consequences

- Server boundary становится меньше и менее чувствительным.
- Backend API, logs, metrics, traces и support workflows должны учитывать privacy boundary.
- Приложению нужны надежные local persistence, export/import UX и recovery instructions.
- Пользователь может потерять локальные данные, если не создает и не хранит backup.

## Проработка

- Уточнить UX экспорта, импорта и восстановления после несовместимой версии backup.
- Описать правила миграции backup format между версиями приложения.
