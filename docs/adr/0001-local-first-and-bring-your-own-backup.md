# ADR-0001: Local-first and Bring Your Own Backup

Status: Accepted

## Context

Pulse Check может использоваться с пользовательскими названиями, заметками, полями, атрибутами и потенциально сведениями о других людях. Продукт должен минимизировать доступ сервера к этим данным и сохранять понятную границу ответственности: пользователь сам решает, что хранить локально и когда делать export/import.

Серверная часть нужна для аутентификации, координации и агрегированной статистики, но не должна становиться основным хранилищем user-owned data. Это решение задает privacy boundary для последующих ADR, включая ADR-0003 и ADR-0005.

## Decision

Pulse Check строится как local-first приложение. User-owned data по умолчанию остаются на устройстве пользователя. Backup следует модели Bring Your Own Backup: пользователь явно экспортирует зашифрованный backup, хранит его самостоятельно и импортирует при необходимости.

Ключевые правила:

- локальное хранилище пользователя является источником истины для user-owned data;
- сервер по умолчанию не хранит пользовательские заметки, атрибуты, произвольные поля и персональные маппинги;
- export/import запускается явным действием пользователя;
- backup archive должен быть переносимым между устройствами пользователя;
- integrity backup проверяется до импорта.

Планируемый backup archive содержит:

- `manifest.json` - версия формата, версия приложения и metadata backup;
- `data.json` - сериализованные локальные данные пользователя;
- `attachments/` - опциональные вложения;
- `checksums.json` - контрольные суммы файлов архива.

Для ZIP планируется `fflate`, для валидации - Zod, для checksums - Web Crypto API/SHA-256.

## Ограничения

- Browser storage persistence помогает, но не заменяет user-owned backup.
- Пользователь отвечает за хранение экспортированного backup вне приложения.
- Серверные исключения из local-first границы должны быть описаны отдельными ADR.
- Backup должен быть зашифрован до передачи пользователем во внешнее хранилище.
- Импорт должен валидировать структуру и checksums до изменения локальных данных.

## Consequences

- Default server boundary становится меньше и менее чувствительным.
- Приложение должно инвестировать в local persistence, export/import UX, backup integrity и понятные recovery instructions.
- Пользователь может потерять локальные данные, если не создает и не хранит backup.
- Product features, которым нужны shared readable data, требуют отдельного explicit architecture decision.
- Privacy boundary становится базовым ограничением для backend API, logs, metrics, traces и support workflows.

## Проработка

- Спроектировать точный формат `manifest.json`, `data.json` и `checksums.json`.
- Выбрать схему шифрования backup и UX хранения ключа или пароля.
- Спроектировать conflict resolution для импорта backup поверх существующих локальных данных.
