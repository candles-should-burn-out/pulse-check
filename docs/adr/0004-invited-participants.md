# ADR-0004: Приглашенные участники и агрегированная статистика

Status: In Progress

## Context

Pulse Check должен поддержать общий рабочий контекст, в котором пользователь приглашает других людей, а они используют тот же набор статусов для своих локальных сущностей. При этом privacy boundary из ADR-0001 сохраняется: сервер не становится хранилищем пользовательских сущностей, notes, attributes и персональных маппингов. Исключение этого ADR - сервер хранит имена участников, которые нужны для узнаваемости людей внутри shared `status_set`.

ADR-0003 вводит server-managed `status_set` и read-only режим для участников. Этот ADR уточняет модель приглашений, дерево участников и агрегированную статистику, которую можно безопасно хранить на backend без раскрытия конкретных локальных данных.

Решение еще не реализовано и фиксирует целевое поведение для будущей разработки.

## Decision

Ввести модель приглашенных участников. Пользователь создает ссылку приглашения или QR-код, передает ее другому человеку, а принявший приглашение пользователь получает `status_set` пригласившего.

Термины:

- "владелец" - пользователь, которому принадлежит `status_set`;
- "пригласивший" - пользователь, создавший приглашение в своем `status_set`;
- "участник" - пользователь, принявший приглашение и использующий `status_set` пригласившего.
- `self_name` - обязательное имя, которое участник сам указывает для отображения в Pulse Check; хранится на backend в membership и видно пользователям, которым доступен этот участник.
- `alias_name` - локальное имя, которое пользователь задает для другого доступного ему участника; хранится только на frontend, не передается на backend и используется как основной label в UI.

Ключевые правила:

- продуктовый термин для UI - "участник";
- каждый пользователь может иметь только один `status_set`;
- участник наследует `status_set` пригласившего пользователя;
- участник может просматривать статусы, но не может создавать, редактировать или удалять их;
- участник может приглашать других пользователей в тот же `status_set`;
- ссылка приглашения создается backend-ом как одноразовый token с ограниченным сроком жизни;
- QR-код является frontend-представлением ссылки приглашения и не требует отдельной backend-сущности;
- ссылка приглашения передает token во frontend fragment, например `/join#token=...`, чтобы token не попадал в backend path, query string и access logs;
- backend хранит только hash token-а, а plaintext token возвращает только при создании приглашения;
- первый вход приглашенного пользователя проходит через Keycloak invite-scoped registration/login flow;
- после успешного login frontend вызывает backend для принятия приглашения текущим authenticated user;
- backend создает связь участника с `status_set` пригласившего;
- backend хранит агрегированную статистику по `status_id` и `user_id`;
- владелец видит агрегированную статистику всех участников своего `status_set`;
- участник видит собственную агрегированную статистику и агрегированную статистику своей ветки приглашений;
- список участников возвращается backend-ом с техническими идентификаторами, связями приглашений и `self_name` участника;
- участник должен указать непустое собственное отображаемое имя, которое хранится на backend в рамках его membership;
- Keycloak profile name не является источником истины для participant `self_name` в рамках этого ADR; backend хранит product-scoped имя участника, а Keycloak остается источником истины для authentication;
- пользователь может задать локальный `alias_name` для доступного ему участника, чтобы видеть этого участника под удобным для себя именем;
- если `alias_name` задан, frontend показывает `alias_name` основным текстом, а `self_name` участника показывает во всплывающем окошке при наведении;
- если `alias_name` не задан, frontend показывает `self_name` участника;
- владелец и участник с правом на свою ветку могут удалить участника полностью из `status_set`;
- полное удаление участника удаляет его связь, ветку приглашенных им участников и их агрегированную статистику из этого `status_set`;
- владелец и участник с правом на свою ветку могут заменить участника другим существующим участником из того же `status_set`;
- при замене статистика заменяемого участника удаляется, его прямые дочерние участники становятся дочерними участниками заменившего пользователя, а membership заменяемого участника удаляется;
- активные приглашения, созданные заменяемым участником, отзываются при замене;
- replacement user должен быть активным участником того же `status_set`, не должен совпадать с заменяемым участником и не должен находиться в ветке заменяемого участника;
- замена не должна нарушать `PARTICIPANT_TREE_MAX_DEPTH`;
- владелец и участник с правом на свою ветку могут заблокировать участника без удаления его связи, ветки приглашений и агрегированной статистики;
- заблокированный участник остается в списках и статистике, но его учетная запись блокируется в Keycloak и он не может войти в аккаунт;
- backend сохраняет состояние блокировки в membership для аудита и дополнительно отклоняет product API запросы от уже выданных token-ов заблокированного участника;
- удаление или блокировка участника не удаляет его учетную запись в Keycloak.

Источники истины:

- Keycloak является источником истины для учетных записей, login, password, account-level блокировок и password reset;
- backend является источником истины для приглашений, membership graph, participant-level блокировок, `self_name` участников и агрегированных счетчиков;
- локальное хранилище frontend является источником истины для `alias_name` участников;
- локальное хранилище пользователя остается источником истины для конкретных сущностей, notes, attributes и персональных маппингов.

## User-flow

### New Flow: Создание приглашения

Потребитель: authenticated user.

Пользователь открывает страницу участников, создает приглашение, получает ссылку и при необходимости показывает QR-код. Frontend отображает plaintext token только в момент создания приглашения и не сохраняет его на backend.

### New Flow: Принятие приглашения

Потребитель: anonymous user, authenticated user.

Пользователь открывает `/join#token=...`. Frontend читает token из URL fragment, сохраняет его в session storage и проверяет приглашение через backend. Anonymous user проходит invite-scoped registration/login flow в Keycloak, возвращается на `/join` и принимает приглашение. Authenticated user принимает приглашение сразу, если его аккаунт еще не имеет `status_set`.

### New Flow: Управление участниками

Потребитель: authenticated user.

Владелец видит всех участников своего `status_set`. Участник видит себя и участников из своей ветки приглашений. Пользователь может задать свой `self_name`, назначать локальные `alias_name` для доступных ему участников, отзывать созданные им приглашения, блокировать, разблокировать, заменять и полностью удалять доступных ему участников в рамках правил доступа.

### New Flow: Имена участников

Потребитель: authenticated user.

Участник задает обязательный непустой `self_name`, который хранится на backend и показывается пользователям, которым доступен этот участник. Пользователь, который видит участника в своей ветке, может задать локальный `alias_name` для этого участника. `alias_name` хранится только на frontend и не передается на backend. Если `alias_name` задан, UI показывает `alias_name` основным текстом, а `self_name` участника показывает в tooltip при наведении.

### New Flow: Блокировка и полное удаление участника

Потребитель: authenticated user.

Пользователь выбирает участника из доступной ветки. Блокировка сохраняет membership, ветку приглашений и агрегированную статистику, но блокирует учетную запись участника в Keycloak и не дает ему войти в аккаунт. Backend также отклоняет product API запросы от уже выданных token-ов заблокированного участника. Полное удаление удаляет membership выбранного участника, всю его ветку приглашений и агрегированную статистику этой ветки.

### New Flow: Замена участника

Потребитель: authenticated user.

Пользователь выбирает заменяемого участника и replacement user из существующих участников того же `status_set`. Backend атомарно удаляет агрегированную статистику заменяемого участника, переносит его прямых дочерних участников на replacement user, удаляет membership заменяемого участника и отзывает активные приглашения, созданные заменяемым участником. Replacement user сохраняет свою статистику и `self_name`. Локальные `alias_name` на frontend пересчитываются по оставшимся `user_id`. Операция запрещена, если replacement user совпадает с заменяемым участником, находится в его ветке или перенос нарушит лимит глубины дерева.

### Exist Flow: Управление статусами

Потребитель: authenticated user.

Для владельца flow управления статусами не меняется. Для участника существующие страницы статусов открываются в read-only режиме: участник использует статусы владельца, но не меняет их определения.

## Database

Backend хранит приглашения, дерево участников и агрегированные счетчики в PostgreSQL.

### Таблица `status_set_memberships`

Хранит связь пользователя с `status_set`.

| Поле                     | Тип           | Constraint | Ограничения                                                                                                                                                                       |
|--------------------------|---------------|------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `user_id`                | `TEXT`        | not null   | Первичный ключ; один набор статусов на аккаунт                                                                                                                                    |
| `status_set_id`          | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `status_set_memberships_status_set_id_idx`; часть уникального индекса `status_set_memberships_status_set_user_idx` |
| `owner_user_id`          | `TEXT`        | not null   | Идентификатор владельца набора статусов                                                                                                                                           |
| `invited_by_user_id`     | `TEXT`        | null       | Идентификатор пригласившего пользователя; `null` для владельца                                                                                                                    |
| `accepted_invitation_id` | `UUID`        | null       | Внешний ключ на `participant_invitations.id`; `null` для владельца                                                                                                                |
| `self_name`              | `TEXT`        | not null   | `self_name` участника; trimmed; от 1 до 40 символов                                                                                                                               |
| `blocked_at`             | `TIMESTAMPTZ` | null       | Дата блокировки участника в этом `status_set`; `null`, если участник активен                                                                                                      |
| `blocked_by_user_id`     | `TEXT`        | null       | Идентификатор пользователя, который заблокировал участника; `null`, если участник активен                                                                                         |
| `created_at`             | `TIMESTAMPTZ` | not null   | Дата создания связи, выставляется backend-ом                                                                                                                                      |

Дополнительные заметки:

- `status_set_memberships.user_id` - если удаленный участник снова входит в продукт без нового приглашения, backend создает для него собственный `status_set` как для нового владельца.
- `status_set_memberships.invited_by_user_id` - формирует дерево приглашений внутри одного `status_set`; значение должно ссылаться на пользователя из того же `status_set`.
- `status_set_memberships.accepted_invitation_id` - нужно для аудита принятия приглашения и диагностики конфликтов.
- `status_set_memberships.self_name` - имя, которое участник указал о себе сам; оно видно пользователям, которым доступен этот участник; пустая строка после trim запрещена.
- `status_set_memberships.blocked_at` - блокировка синхронизируется с Keycloak account disable и запрещает product API для уже выданных token-ов, но не удаляет membership, дерево приглашений и агрегированную статистику.
- `status_set_memberships.blocked_by_user_id` - нужно для аудита; значение должно ссылаться на пользователя, которому было доступно управление участником на момент блокировки.
- `status_set_memberships_status_set_user_idx` нужен для внешних ключей агрегированной статистики.

### Таблица `participant_invitations`

Хранит invitation lifecycle.

| Поле                 | Тип           | Constraint | Ограничения                                                                                                |
|----------------------|---------------|------------|------------------------------------------------------------------------------------------------------------|
| `id`                 | `UUID`        | not null   | Первичный ключ, генерируется backend-ом                                                                    |
| `status_set_id`      | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `participant_invitations_status_set_id_idx` |
| `owner_user_id`      | `TEXT`        | not null   | Идентификатор владельца набора статусов                                                                    |
| `invited_by_user_id` | `TEXT`        | not null   | Идентификатор пользователя, создавшего приглашение                                                         |
| `token_hash`         | `TEXT`        | not null   | Уникальный hash token-а; plaintext token не хранится                                                       |
| `expires_at`         | `TIMESTAMPTZ` | not null   | Дата истечения срока действия                                                                              |
| `revoked_at`         | `TIMESTAMPTZ` | null       | Дата ручного отзыва приглашения                                                                            |
| `accepted_at`        | `TIMESTAMPTZ` | null       | Дата успешного принятия одноразового приглашения; `null`, если приглашение еще не использовано             |
| `created_at`         | `TIMESTAMPTZ` | not null   | Дата создания записи, выставляется backend-ом                                                              |

Дополнительные заметки:

- `participant_invitations.token_hash` - вычисляется через cryptographic hash от случайного token-а с достаточной entropy.
- `participant_invitations.revoked_at` - отозванное приглашение нельзя принять, даже если срок действия не истек.
- `participant_invitations.accepted_at` - после успешного принятия приглашение считается использованным и не может быть принято повторно.

### Таблица `entity_status_counters`

Хранит агрегаты локальных сущностей по статусам.

| Поле            | Тип           | Constraint | Ограничения                                                                                                                                         |
|-----------------|---------------|------------|-----------------------------------------------------------------------------------------------------------------------------------------------------|
| `status_set_id` | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; часть составного первичного ключа и внешнего ключа на `status_set_memberships`              |
| `user_id`       | `TEXT`        | not null   | Идентификатор пользователя, которому принадлежат локальные сущности; часть составного первичного ключа и внешнего ключа на `status_set_memberships` |
| `status_id`     | `UUID`        | not null   | Внешний ключ на `statuses.id` с `ON DELETE CASCADE`; индекс `entity_status_counters_status_id_idx`; часть составного первичного ключа               |
| `count`         | `INTEGER`     | not null   | Количество локальных сущностей пользователя в этом статусе; не меньше 0                                                                             |
| `updated_at`    | `TIMESTAMPTZ` | not null   | Дата последнего обновления агрегата, выставляется backend-ом                                                                                        |

Дополнительные заметки:

- Первичный ключ `entity_status_counters` состоит из `status_set_id`, `user_id`, `status_id`.
- Внешний ключ `status_set_id`, `user_id` ссылается на `status_set_memberships.status_set_id`, `status_set_memberships.user_id` с `ON DELETE CASCADE`.
- `entity_status_counters` хранит только агрегаты и не содержит идентификаторы конкретных сущностей.
- При полном удалении участника backend удаляет его строки `entity_status_counters` вместе со строками удаляемой ветки участников.
- При блокировке участника backend сохраняет его строки `entity_status_counters`.

### Таблица `participant_replacements`

Хранит audit trail разрушительной операции замены участника.

| Поле                        | Тип           | Constraint | Ограничения                                                                                                 |
|-----------------------------|---------------|------------|-------------------------------------------------------------------------------------------------------------|
| `id`                        | `UUID`        | not null   | Первичный ключ, генерируется backend-ом                                                                     |
| `status_set_id`             | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `participant_replacements_status_set_id_idx` |
| `replaced_user_id`          | `TEXT`        | not null   | Идентификатор участника, который был заменен и удален из membership graph                                   |
| `replacement_user_id`       | `TEXT`        | not null   | Идентификатор существующего участника, который получил дочернюю ветку замененного участника                 |
| `replaced_by_user_id`       | `TEXT`        | not null   | Идентификатор пользователя, который выполнил замену                                                         |
| `reassigned_children_count` | `INTEGER`     | not null   | Количество прямых дочерних участников, перенесенных на replacement user                                     |
| `deleted_counters_count`    | `INTEGER`     | not null   | Количество строк агрегированной статистики, удаленных у замененного участника                               |
| `created_at`                | `TIMESTAMPTZ` | not null   | Дата замены, выставляется backend-ом                                                                        |

Дополнительные заметки:

- `participant_replacements` не содержит имен участников, notes, attributes или конкретных локальных сущностей.
- `replacement_user_id` должен ссылаться на активную membership на момент операции, но запись audit trail сохраняет id даже если replacement user будет удален позже.

## API

Все product API ручки, кроме проверки ссылки приглашения, требуют authenticated user из Bearer JWT, если OIDC включен. Если membership текущего пользователя заблокирован, backend возвращает `403 Forbidden` и не выполняет product API действие для уже выданного token-а. Новые login attempts заблокированного участника должны отклоняться Keycloak-ом, потому что backend при блокировке отключает его учетную запись.

Ошибка заблокированного участника:

Код ответа: `403 Forbidden`

```json
{
  "error": "participant_blocked"
}
```

### POST: /participant-invitations

Тело запроса:

```json
{
  "expires_in_hours": 48
}
```

Код ответа: `201 Created`

Ответ:

```json
{
  "id": "7d2a6f2b-3d4e-4f6a-9a8b-111111111111",
  "invite_url": "https://pulse-check.example/join#token=plaintext-token",
  "token": "plaintext-token",
  "expires_at": "2026-05-23T10:00:00Z",
  "accepted_at": null,
  "created_at": "2026-05-16T10:00:00Z"
}
```

Права доступа: доступно владельцу и участнику. Приглашение создается в `status_set` текущего пользователя. Ссылка приглашения всегда одноразовая.

Ошибки:

Код ответа: `400 Bad Request`

```json
{
  "error": "invalid_invitation"
}
```

Код ответа: `409 Conflict`

```json
{
  "error": "participant_limit_reached"
}
```

### POST: /participant-invitations/validate

Тело запроса:

```json
{
  "token": "plaintext-token"
}
```

Код ответа: `200 OK`

Ответ:

```json
{
  "valid": true,
  "expires_at": "2026-05-23T10:00:00Z"
}
```

Права доступа: доступно анонимному и authenticated user. Ответ не раскрывает `owner_user_id`, `invited_by_user_id`, имена пользователей или статистику.

Ошибки:

Код ответа: `404 Not Found`

```json
{
  "error": "invitation_not_found"
}
```

Код ответа: `410 Gone`

```json
{
  "error": "invitation_expired"
```

## Страницы

### /join

Назначение: принять приглашение по ссылке или QR-коду.

Доступ: anonymous user и authenticated user. Anonymous user сначала проходит invite-scoped registration/login flow в Keycloak, затем возвращается на страницу принятия приглашения. Token приглашения читается frontend-ом из URL fragment и не отправляется в backend path или query string.

Основные элементы:

- состояние проверки ссылки приглашения;
- действие входа или регистрации через Keycloak;
- обязательное поле `self_name` участника;
- подтверждение принятия приглашения;
- сообщение об истекшей или отозванной ссылке;
- сообщение о конфликте, если аккаунт уже привязан к другому `status_set`.

Действия:

- открыть ссылку приглашения - frontend читает token из URL fragment, сохраняет его в session storage и вызывает `POST /participant-invitations/validate`;
- войти или зарегистрироваться - пользователь проходит Keycloak flow и возвращается на `/join`;
- принять приглашение - frontend отправляет `POST /participant-invitations/accept` с token и непустым `self_name`;
- перейти в приложение - после успешного принятия пользователь попадает в `/app`.

Состояния:

- загрузка проверки token-а;
- приглашение действительно;
- приглашение истекло, отозвано или уже использовано;
- пользователь не authenticated;
- аккаунт уже имеет `status_set`;
- `self_name` пустой или невалидный;
- успешное принятие приглашения;
- ошибка принятия приглашения.

### /app/participants

Назначение: управлять приглашениями, именами участников, `alias_name`, заменой, блокировкой и полным удалением участников.

Доступ: authenticated user. Владелец видит всех участников своего `status_set`. Участник видит себя и участников из своей ветки приглашений.

Основные элементы:

- список участников, доступных текущему пользователю;
- `self_name` участника, хранящийся на backend;
- локальный `alias_name` участника, хранящийся только на frontend и видимый только текущему пользователю;
- tooltip с `self_name`, если основным текстом показан `alias_name`;
- техническое состояние участника без отображения backend-owned персональных данных;
- признак блокировки участника;
- кнопка создания ссылки приглашения;

- действие отзыва приглашения;
- действие изменения собственного `self_name`;
- действие задания, изменения и удаления локального `alias_name`;
- действие замены участника существующим участником;
- действие блокировки и разблокировки участника;

- создать приглашение - frontend отправляет `POST /participant-invitations`, показывает ссылку и генерирует QR-код локально;

### /app/statistics/statuses

Назначение: просматривать агрегированную статистику по статусам, участникам и суммарно по доступной ветке приглашений.

Доступ: authenticated user. Владелец видит всех участников своего `status_set`. Участник видит себя и участников из своей ветки приглашений.

Основные элементы:

- агрегированная статистика по статусам текущего пользователя;
- агрегированная статистика по каждому доступному участнику;
- суммарная статистика владельца и доступных участников;
- `self_name` участников из backend;
- локальные `alias_name` из frontend storage;

Действия:

- просмотреть статистику по каждому участнику;
- переключить режим отображения между разрезом по участникам и суммарным видом;
- отфильтровать статистику по участнику;
- обновить агрегаты после локального изменения сущностей текущего пользователя через `PUT /statistics/status-counters`.

- загрузка агрегатов;

- участник удален и больше не используется в списке статистики;
- участник заблокирован и его сохраненная статистика остается доступной;
- read-only режим без действий изменения чужой статистики;
- отсутствие раскрытия конкретных сущностей, notes, attributes и персональных маппингов.

## Ограничения

- Ссылка приглашения по умолчанию живет 48 часов.
- Ссылка приглашения всегда одноразовая.
- QR-код является представлением одноразовой ссылки и не поддерживает многоразовое принятие.
- Максимальная глубина дерева приглашений по умолчанию равна 5.
- Максимальное количество участников в одном `status_set` по умолчанию равно 100.
- Участник не может менять определения статусов.
- Участник не может принять приглашение, если его аккаунт уже имеет `status_set`.
- Заблокированный участник не может войти в аккаунт через Keycloak.
- Backend дополнительно отклоняет product API запросы заблокированного участника, если у него остался ранее выданный token.
- Заблокированный участник остается в membership graph и учитывается в лимите участников до полного удаления.
- Блокировка участника отключает его учетную запись в Keycloak, но не удаляет учетную запись и не удаляет его локальные данные на устройстве.
- Полное удаление участника необратимо удаляет его membership, ветку приглашений и агрегированную статистику этой ветки из `status_set`.
- Замена участника необратимо удаляет membership и агрегированную статистику заменяемого участника, но сохраняет его прямую дочернюю ветку через reparenting на replacement user.
- Replacement user не может находиться в ветке заменяемого участника, чтобы не создать цикл в membership graph.
- Replacement user должен быть активным участником того же `status_set`.
- Замена участника не должна приводить к превышению `PARTICIPANT_TREE_MAX_DEPTH`.
- Backend хранит только обязательный `self_name` участников; это имя является частью shared participant UX и не должно использоваться как источник identity для authentication.
- `self_name` ограничен 40 символами после trim.
- `alias_name` хранится только локально на frontend и не передается на backend.
- Backend не хранит конкретные сущности, notes, attributes или персональные маппинги.
- Backend не должен логировать plaintext invitation token.
- Полное удаление участника из Pulse Check не удаляет учетную запись в Keycloak.
- Invite-scoped registration/login требует настройки Keycloak и не должен включать публичную self-registration без token-а приглашения.

## Consequences

- Пользователи могут расширять общий рабочий контекст через ссылки и QR-коды.
- Статистика владельца и участников остается сопоставимой, потому что все используют один `status_set`.
- Участники могут строить собственные ветки приглашений без прав на редактирование статусов.
- Backend получает ответственность за invitation lifecycle, membership graph и агрегированные счетчики.
- Backend должен синхронизировать participant-level блокировку с Keycloak account disable и проверять блокировку во всех product API ручках для защиты от ранее выданных token-ов.
- Backend получает ответственность за server-side participant `self_name`, поэтому privacy boundary становится шире, чем в варианте с полностью локальными именами.
- Frontend получает ответственность за QR-код, clipboard UX, tooltip для `self_name` при наличии `alias_name` и отображение статистики в разрезах.
- Frontend получает ответственность за хранение `alias_name` и его очистку после полного удаления или замены участника.
- Frontend должен явно различать блокировку и полное удаление, потому что блокировка обратима и сохраняет статистику, а удаление необратимо чистит ветку.
- Frontend должен явно различать замену и удаление: замена сохраняет дочернюю ветку через reparenting, удаление удаляет всю ветку.
- Keycloak остается ответственным за учетные записи, пароль и account-level блокировки, поэтому Pulse Check не реализует хранение паролей.
- Privacy boundary сохраняется для конкретных сущностей, notes, attributes, `alias_name` и персональных маппингов, но `self_name` участников становится server-managed данными.
- Блокировка участника сохраняет агрегированную статистику и историю связи в `status_set`.
- Замена участника теряет статистику заменяемого участника, но позволяет сохранить подчиненную ветку при организационной замене человека.
- Полное удаление участника необратимо удаляет его агрегированную статистику в данном `status_set`, но не затрагивает его локальные данные на устройстве.

## Проработка

- Спроектировать точный Keycloak invite-scoped registration/login flow: extension, theme, required actions или другой механизм, который разрешает регистрацию только по token-у приглашения.
- Спроектировать UX подтверждения блокировки и полного удаления участника с отображением разных последствий и количества участников в удаляемой ветке.
- Спроектировать UX подтверждения замены участника: показать потерю статистики заменяемого участника, количество переносимых прямых детей и имя replacement user.
- Спроектировать отдельный flow агрегированной статистики и определить, как frontend синхронизирует `PUT /statistics/status-counters`: при каждом изменении сущностей, debounce, manual sync или background sync.
- Спроектировать миграцию для существующих пользователей: создание self-membership и перенос текущих статусов в собственный `status_set`.
