# ADR-0005: Серверно управляемые наборы статусов

Date: 2026-05-15

Status: Accepted

## Context

Pulse Check использует статусы для учета сущностей и агрегированной статистики. Модель помощников требует, чтобы приглашенный пользователь использовал тот же набор статусов, что и пригласивший пользователь.

Для стабильной статистики статусам нужны неизменяемые технические идентификаторы. Для понятного UI статусам также нужны пользовательское имя и цвета, которые должны одинаково отображаться у владельца и помощников.

Это решение уточняет границу приватности из ADR-0001 и ADR-0003: имя и цвета статуса являются осознанным исключением из правила "пользовательские статусы локальны по умолчанию", потому что они нужны для совместного сценария работы и агрегированной статистики.

## Decision

Ввести серверно управляемый набор статусов, принадлежащий пользователю-создателю. Backend является источником истины для набора статусов.

Ключевые правила:

- каждый пользователь-владелец имеет один status set;
- каждый аккаунт имеет один эффективный status set;
- `status_id` является неизменяемым UUID, который генерирует backend;
- владелец набора статусов может создавать, редактировать и удалять статусы;
- помощник использует status set пригласившего пользователя в режиме только для чтения;
- роль пользователя вычисляется из связи `status_set_memberships`: если `user_id` равен `owner_user_id`, пользователь является владельцем, иначе помощником;
- предустановленные палитры статусов в v1 являются конфигурацией frontend и не требуют отдельной backend-ручки;
- backend валидирует и хранит итоговые цвета, выбранные пользователем из палитры или собранные вручную.

Пагинации для статусов не будет. Максимальное количество статусов в одном наборе по умолчанию равно 50 и должно настраиваться через backend configuration `STATUS_SET_MAX_STATUSES`.

## Данные и База Данных

Backend хранит наборы статусов в PostgreSQL.

Таблица `status_sets`:

| Поле            | Тип           | Constraint | Ограничения                                                 |
|-----------------|---------------|------------|-------------------------------------------------------------|
| `id`            | `UUID`        | not null   | Первичный ключ, генерируется backend-ом                     |
| `owner_user_id` | `TEXT`        | not null   | Уникальный идентификатор владельца из контекста авторизации |
| `created_at`    | `TIMESTAMPTZ` | not null   | Дата создания записи, выставляется backend-ом               |
| `updated_at`    | `TIMESTAMPTZ` | not null   | Дата последнего изменения записи, обновляется backend-ом    |

Таблица `status_set_memberships`:

| Поле            | Тип           | Constraint | Ограничения                                                                                               |
|-----------------|---------------|------------|-----------------------------------------------------------------------------------------------------------|
| `user_id`       | `TEXT`        | not null   | Первичный ключ; один эффективный набор статусов на аккаунт                                                |
| `status_set_id` | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `status_set_memberships_status_set_id_idx` |
| `owner_user_id` | `TEXT`        | not null   | Идентификатор владельца эффективного набора статусов                                                      |
| `created_at`    | `TIMESTAMPTZ` | not null   | Дата создания связи, выставляется backend-ом                                                              |

Таблица `statuses`:

| Поле               | Тип           | Constraint | Ограничения                                                                                 |
|--------------------|---------------|------------|---------------------------------------------------------------------------------------------|
| `id`               | `UUID`        | not null   | Первичный ключ, генерируется backend-ом и не меняется                                       |
| `status_set_id`    | `UUID`        | not null   | Внешний ключ на `status_sets.id` с `ON DELETE CASCADE`; индекс `statuses_status_set_id_idx` |
| `name`             | `TEXT`        | not null   | Не длиннее 40 символов; check constraint `char_length(name) <= 40`                          |
| `border_color`     | `CHAR(7)`     | not null   | HEX-цвет в формате `#RRGGBB`; check constraint по regex `^#[0-9A-Fa-f]{6}$`                 |
| `background_color` | `CHAR(7)`     | not null   | HEX-цвет в формате `#RRGGBB`; check constraint по regex `^#[0-9A-Fa-f]{6}$`                 |
| `text_color`       | `CHAR(7)`     | not null   | HEX-цвет в формате `#RRGGBB`; check constraint по regex `^#[0-9A-Fa-f]{6}$`                 |
| `created_at`       | `TIMESTAMPTZ` | not null   | Дата создания записи, выставляется backend-ом                                               |
| `updated_at`       | `TIMESTAMPTZ` | not null   | Дата последнего изменения записи, обновляется backend-ом                                    |

Дополнительные заметки:

- `status_sets.owner_user_id` - один владелец не может иметь больше одного собственного набора статусов.
- `status_set_memberships.user_id` - когда пользователь становится помощником, связь указывает на `status_set_id` пригласившего.
- `statuses.name` - ограничение 40 символов проверяется на frontend для мгновенной обратной связи и на backend как source of truth.
- `statuses.*_color` - backend хранит только итоговые цвета, а не идентификатор frontend-палитры.

## API

Все ручки статусов требуют authenticated user из Bearer JWT, если OIDC включен.

### GET: /status-set

Тело запроса: отсутствует

Код ответа: `200 OK`

Ответ:

```json
{
  "id": "7d2a6f2b-3d4e-4f6a-9a8b-111111111111",
  "owner_user_id": "owner-user-id",
  "role": "status_owner",
  "statuses": [
    {
      "id": "f3a5d6e7-8b9c-4d10-9e11-222222222222",
      "name": "Активен",
      "border_color": "#2F6BFF",
      "background_color": "#EAF0FF",
      "text_color": "#102A6B",
      "created_at": "2026-05-15T10:00:00Z",
      "updated_at": "2026-05-15T10:00:00Z"
    }
  ]
}
```

Права доступа: владелец получает `role: "status_owner"`, помощник получает `role: "assistant"`.

### GET: /statuses

Тело запроса: отсутствует

Код ответа: `200 OK`

Ответ:

```json
[
  {
    "id": "f3a5d6e7-8b9c-4d10-9e11-222222222222",
    "name": "Активен",
    "border_color": "#2F6BFF",
    "background_color": "#EAF0FF",
    "text_color": "#102A6B",
    "created_at": "2026-05-15T10:00:00Z",
    "updated_at": "2026-05-15T10:00:00Z"
  }
]
```

Права доступа: владелец и помощник получают полный список статусов из эффективного набора без пагинации.

### POST: /statuses

Тело запроса:

```json
{
  "name": "Активен",
  "border_color": "#2F6BFF",
  "background_color": "#EAF0FF",
  "text_color": "#102A6B"
}
```

Код ответа: `201 Created`

Ответ:

```json
{
  "id": "f3a5d6e7-8b9c-4d10-9e11-222222222222",
  "name": "Активен",
  "border_color": "#2F6BFF",
  "background_color": "#EAF0FF",
  "text_color": "#102A6B",
  "created_at": "2026-05-15T10:00:00Z",
  "updated_at": "2026-05-15T10:00:00Z"
}
```

Права доступа: доступно только владельцу набора статусов. Для помощника возвращается `403 Forbidden`.

Ошибки:

Код ответа: `400 Bad Request`

```json
{
  "error": "invalid_status"
}
```

Код ответа: `403 Forbidden`

```json
{
  "error": "status_set_read_only"
}
```

Код ответа: `409 Conflict`

```json
{
  "error": "status_limit_reached"
}
```

Права доступа: доступно только владельцу набора статусов. `id` статуса не меняется. Для помощника возвращается `403 Forbidden`.

## Страницы

### /app/statuses

Назначение: управление эффективным набором статусов текущего пользователя.

Доступ: authenticated user. Владелец набора статусов видит режим управления. Помощник видит унаследованный набор статусов пригласившего пользователя в режиме только для чтения.

Основные элементы:

- список статусов из эффективного набора текущего пользователя;
- форма создания статуса для владельца;
- форма редактирования имени и цветов статуса для владельца;
- выбор предустановленной frontend-палитры;
- ручной выбор `border_color`, `background_color` и `text_color`;
- визуальный preview статуса с выбранными цветами;
- read-only представление унаследованных статусов для помощника.

Действия:

- создать статус - владелец добавляет новый статус в свой набор, frontend отправляет `POST /statuses` и показывает созданный статус с backend-generated `id`;
- изменить статус - владелец обновляет `name`, `border_color`, `background_color` и `text_color`, frontend отправляет `PATCH /statuses/{status_id}` и сохраняет неизменный `id`;
- удалить статус - владелец удаляет статус через `DELETE /statuses/{status_id}`;
- выбрать предустановленную палитру - frontend заполняет итоговые цвета статуса без отдельной backend-ручки;
- собрать палитру вручную - пользователь выбирает итоговые цвета, которые backend валидирует и сохраняет.

Состояния:

- загрузка effective status set;
- пустое состояние, если в наборе еще нет статусов;
- ошибка загрузки или сохранения;
- ошибка валидации имени длиннее 40 символов;
- ошибка валидации HEX-цветов;
- ошибка достижения лимита статусов `status_limit_reached`;
- read-only режим для помощника без действий создания, редактирования и удаления;
- успешное создание, изменение и удаление статуса.

### /app/statistics/statuses

Назначение: будущий экран статистики в разрезе статусов и помощников без раскрытия конкретных сущностей.

Доступ: authenticated user. Экран работает в режиме только для чтения для владельца и помощника.

Основные элементы:

- агрегированная статистика по статусам из эффективного набора пользователя;
- разрез по помощникам, если он доступен текущему пользователю;
- названия и цвета статусов из server-managed status set;
- показатели, построенные только на агрегатах.

Действия:

- просмотреть агрегированную статистику по статусам;
- просмотреть агрегированную статистику по помощникам без перехода к конкретным сущностям.

Состояния:

- загрузка агрегатов;
- пустое состояние, если агрегатов еще нет;
- ошибка загрузки агрегатов;
- read-only режим без действий изменения;
- отсутствие раскрытия конкретных сущностей, имен, notes, attributes и персональных маппингов.

Связанные экраны создания и редактирования сущностей должны использовать только статусы из эффективного набора статусов текущего пользователя. Помощник не должен видеть отдельный локальный набор статусов.

## Ограничения

- Максимальное количество статусов в одном наборе по умолчанию равно 50.
- Лимит статусов настраивается через `STATUS_SET_MAX_STATUSES`.
- Имя статуса ограничено 40 символами.
- `border_color`, `background_color` и `text_color` должны быть HEX-цветами в формате `#RRGGBB`.
- Пагинации для статусов не будет.
- Помощник не может создавать, редактировать или удалять статусы.
- Пользователь не должен использовать в названиях статусов персональные, чувствительные, оскорбительные, дискриминационные или специальные категории персональных данных.
- Экран `/app/statistics/statuses` должен строиться на агрегатах и не раскрывать конкретные сущности.

## Consequences

- `status_id` становится стабильным ключом для сущностей и агрегатов даже при изменении имени или цветов статуса.
- Владелец и помощники видят один и тот же набор статусов, поэтому статистика по статусам остается сопоставимой между участниками.
- API статусов остается простым и возвращает полный справочник без контракта пагинации; масштабирование ограничивается настраиваемым лимитом.
- Backend начинает хранить пользовательские имена и цвета статусов, поэтому они должны соответствовать ограничениям минимизации данных.
- PostgreSQL становится продуктовым хранилищем для определений статусов и связей с эффективным набором статусов; это отдельное хранилище от собственной базы пользователей Keycloak.
- Удаление используемого статуса временно может оставлять сущности без валидного статуса до отдельной доработки поведения очистки или переназначения.

## Проработка

- Спроектировать сценарий удаления или переназначения для статуса, который уже используется сущностями: какие сущности требуют нового статуса, кто выполняет переназначение и как показывать это в UI.
- Спроектировать экран `/app/statistics/statuses` для статистики в разрезе статусов и помощников без раскрытия конкретных сущностей.
