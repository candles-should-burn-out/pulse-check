# ADR-0005: Серверно управляемые наборы статусов

Date: 2026-05-15

Status: Accepted

## Context

Pulse Check использует статусы для учета сущностей и агрегированной статистики. Модель помощников требует, чтобы приглашенный пользователь использовал тот же набор статусов, что и пригласивший пользователь.

Для стабильной статистики статусам нужны неизменяемые технические идентификаторы. Для понятного UI статусам также нужны пользовательское имя и цвета, которые должны одинаково отображаться у владельца и помощников.

## Decision

Ввести серверно управляемый набор статусов, принадлежащий пользователю-создателю. Backend является источником истины для набора статусов и хранит для каждого статуса:

- `id` - неизменяемый UUID, который генерирует backend.
- `name` - имя, изменяемое создателем статуса.
- `border_color`, `background_color`, `text_color` - цвета контура, фона и шрифта.
- `owner_user_id` - владелец набора статусов.

Пользователь-создатель управляет статусами на отдельном экране: создает новые статусы, редактирует существующие и удаляет ненужные. UI должен предлагать набор предустановленных цветовых палитр и возможность собрать свою палитру из цветов контура, фона и цвета шрифта.

Пользователь, ставший помощником, на уровне аккаунта не управляет собственными статусами и полностью использует набор статусов пригласившего пользователя. Экран управления статусами для помощника должен работать в режиме только для чтения без действий создания, редактирования и удаления.

Продуктовые API для статусов должны обеспечивать UUID, который генерирует backend, сохранение изменений только создателем и запрет операций изменения для помощников.

Backend хранит наборы статусов в PostgreSQL.

Имя статуса ограничивается 40 символами. Ограничение должно проверяться на frontend для мгновенной обратной связи в форме и на backend, потому что API остается источником истины для любых клиентов.

Таблица `status_sets`:

| Поле            | Тип           | Ограничения                                                              |
|-----------------|---------------|--------------------------------------------------------------------------|
| `id`            | `UUID`        | Первичный ключ, генерируется backend-ом                                  |
| `owner_user_id` | `TEXT`        | Обязательно, уникально, идентификатор владельца из контекста авторизации |
| `created_at`    | `TIMESTAMPTZ` | Обязательно                                                              |
| `updated_at`    | `TIMESTAMPTZ` | Обязательно                                                              |

Таблица `status_set_memberships`:

| Поле                 | Тип           | Ограничения                                                            |
|----------------------|---------------|------------------------------------------------------------------------|
| `user_id`            | `TEXT`        | Первичный ключ, один эффективный набор статусов на аккаунт             |
| `status_set_id`      | `UUID`        | Обязательно, ссылка на `status_sets.id`                                |
| `owner_user_id`      | `TEXT`        | Обязательно, владелец набора статусов                                  |
| `created_at`         | `TIMESTAMPTZ` | Обязательно                                                            |

Таблица `statuses`:

| Поле               | Тип           | Ограничения                                           |
|--------------------|---------------|-------------------------------------------------------|
| `id`               | `UUID`        | Первичный ключ, генерируется backend-ом и не меняется |
| `status_set_id`    | `UUID`        | Обязательно, ссылка на `status_sets.id`               |
| `name`             | `TEXT`        | Обязательно, не длиннее 40 символов                   |
| `border_color`     | `CHAR(7)`     | Обязательно, HEX-цвет в формате `#RRGGBB`             |
| `background_color` | `CHAR(7)`     | Обязательно, HEX-цвет в формате `#RRGGBB`             |
| `text_color`       | `CHAR(7)`     | Обязательно, HEX-цвет в формате `#RRGGBB`             |
| `created_at`       | `TIMESTAMPTZ` | Обязательно                                           |
| `updated_at`       | `TIMESTAMPTZ` | Обязательно                                           |

Для `status_set_memberships` роль пользователя вычисляется из сравнения `user_id` и `owner_user_id`: если значения равны, пользователь является владельцем набора статусов; если отличаются, пользователь является помощником. Когда пользователь становится помощником, его связь указывает на `status_set_id` пригласившего, поэтому аккаунт не может управлять отдельным набором статусов, пока эта связь активна.

Обязательные frontend-страницы:

- `/app/statuses` - экран управления статусами. Для владельца показывает список статусов, форму создания, форму редактирования имени и цветов, выбор предустановленной палитры, ручной выбор `border_color`, `background_color`, `text_color` и действие удаления. Для помощника показывает унаследованный набор статусов в режиме только для чтения без действий изменения.
- `/app/statistics/statuses` - будущий экран статистики в разрезе статусов и помощников в режиме только для чтения. Экран должен строиться на агрегатах и не раскрывать конкретные сущности.
- Экраны создания и редактирования сущностей должны использовать только статусы из эффективного набора статусов текущего пользователя. Помощник не должен видеть отдельный локальный набор статусов.

Обязательные backend API:

- `GET /status-set` - возвращает эффективный набор статусов текущего пользователя: владельца набора, роль доступа (`status_owner` или `assistant`) и список статусов.
- `GET /statuses` - возвращает полный список статусов из эффективного набора статусов текущего пользователя без пагинации.
- `POST /statuses` - создает статус в наборе статусов текущего пользователя-владельца. Тело запроса содержит `name`, `border_color`, `background_color`, `text_color`; ответ возвращает созданный статус с `id`, который сгенерировал backend. Для помощника возвращает `403 Forbidden`. При достижении лимита статусов возвращает `409 Conflict` с кодом ошибки `status_limit_reached`.
- `PATCH /statuses/{status_id}` - обновляет `name`, `border_color`, `background_color`, `text_color` существующего статуса без изменения `id`. Доступно только владельцу набора статусов; для помощника возвращает `403 Forbidden`.
- `DELETE /statuses/{status_id}` - удаляет статус из набора статусов владельца. До реализации сценария переназначения удаление статуса, который используется сущностями, разрешено; backend должен записать предупреждение в лог с `status_id` и TODO о том, что у сущностей, использующих этот статус, нужно убрать или переназначить статус. Для помощника возвращает `403 Forbidden`.

Предустановленные палитры статусов в v1 являются конфигурацией frontend и не требуют отдельной backend-ручки. Backend валидирует и хранит итоговые цвета, выбранные пользователем из палитры или собранные вручную.

Пагинации для статусов не будет. Максимальное количество статусов в одном наборе по умолчанию равно 50 и должно настраиваться через конфигурацию backend `STATUS_SET_MAX_STATUSES`.

Это решение уточняет границу приватности из ADR-0001 и ADR-0003: имя и цвета статуса являются осознанным исключением из правила "пользовательские статусы локальны по умолчанию", потому что они нужны для совместного сценария работы и агрегированной статистики.

## Consequences

- `status_id` становится стабильным ключом для сущностей и агрегатов даже при изменении имени или цветов статуса.
- Владелец и помощники видят один и тот же набор статусов, поэтому статистика по статусам остается сопоставимой между участниками.
- API статусов остается простым и возвращает полный справочник без контракта пагинации; масштабирование ограничивается настраиваемым лимитом.
- Backend начинает хранить пользовательские имена и цвета статусов, поэтому они должны соответствовать тем же ограничениям минимизации данных: пользователь не должен использовать в названиях статусов персональные, чувствительные, оскорбительные, дискриминационные или специальные категории персональных данных.
- PostgreSQL становится продуктовым хранилищем для определений статусов и связей с эффективным набором статусов; это отдельное хранилище от собственной базы пользователей Keycloak.
- Удаление используемого статуса временно может оставлять сущности без валидного статуса до отдельной доработки поведения очистки или переназначения.

## Нужно Проработать

- Спроектировать сценарий удаления или переназначения для статуса, который уже используется сущностями: какие сущности требуют нового статуса, кто выполняет переназначение и как показывать это в UI.
- Спроектировать экран `/app/statistics/statuses` для статистики в разрезе статусов и помощников без раскрытия конкретных сущностей.
