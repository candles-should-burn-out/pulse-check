# Чек-лист перед завершением задачи

## 1. Проектирование и ограничения до разработки

- [ ] Если разработка идет по ADR, проверить, что ADR остается актуальной для текущей реализации.
- [ ] Все временные решения, костыли и быстрые фиксы, которые в продакшене нужно исправить, уточнены до разработки.
- [ ] Если временные решения были согласованы и реализованы, они подробно зафиксированы в [todo](../todo/todo.md) как пункты для исправления.
- [ ] Проверено, что новые зависимости действительно необходимы и не дублируют уже используемые библиотеки.
- [ ] Если добавляются новые настройки, env-переменные или feature flags, заранее определено, где они будут описаны в документации.

## 2. Реализация и консистентность кода

- [ ] Публичные интерфейсы, DTO, события, команды и обработчики названы консистентно с существующим кодом.
- [ ] Основные этапы обработки логируются.
- [ ] На уровне `info` логируются только важные этапы обработки.
- [ ] Менее важные детали, необходимые для отладки, логируются с уровнем `debug` или `trace`.
- [ ] Все ошибки имеют понятные сообщения.
- [ ] Ошибки не скрывают важный контекст, необходимый для отладки.
- [ ] Проверено, что ошибки не раскрывают чувствительные данные.

## 3. Внешние зависимости и отказоустойчивость

- [ ] Если фича зависит от внешних сервисов, предусмотрены таймауты.
- [ ] Если фича зависит от внешних сервисов, предусмотрены ретраи, fallback или явно описанное поведение при недоступности зависимости.
- [ ] Проверена идемпотентность операций там, где возможны ретраи или повторные запросы.

## 4. Асинхронная обработка

- [ ] Если добавляется асинхронная обработка, очередь, cron или background job, описано поведение при повторном запуске.
- [ ] Описано поведение при падении процесса или задачи.
- [ ] Описано поведение при дублировании сообщений.
- [ ] Проверена идемпотентность операций, которые могут выполняться повторно.

## 5. База данных и миграции

- [ ] Если меняется схема БД, миграция является обратимой.
- [ ] Если миграция необратимая, описан безопасный план отката.

## 6. API и контракты

- [ ] Все изменения API отражены в [openapi.json](../api/openapi.json).
- [ ] Проверено, что ответы API не раскрывают чувствительные данные.
- [ ] Проверено, что ошибки API имеют понятные сообщения.
- [ ] Проверено, что ошибки API содержат достаточно контекста для диагностики, но не раскрывают чувствительные данные.

## 7. Тесты

- [ ] Написаны тесты.
- [ ] В тестах есть комментарии на русском языке по шагам.
- [ ] Тесты не пытаются тестировать сторонние библиотеки и внешние зависимости.
- [ ] Проверены ошибки, пустые данные, невалидный ввод и неожиданные состояния.
- [ ] Проверена идемпотентность операций там, где возможны ретраи или повторные запросы.
- [ ] Проверено, что чувствительные данные не попадают в тестовые фикстуры.

## 8. Документация

- [ ] Все важные изменения, появившиеся в процессе разработки, зафиксированы в разделе [архитектуры](../architecture).
- [ ] Если добавлены новые настройки, env-переменные или feature flags, они описаны в документации.
- [ ] Если поведение при недоступности внешней зависимости нетривиальное, оно описано в документации.
- [ ] Если есть необратимые миграции, безопасный план отката описан в документации.
- [ ] Если добавлена асинхронная обработка, очередь, cron или background job, поведение при повторном запуске, падении и дублировании сообщений описано в документации.

## 9. Финальная проверка безопасности и качества

- [ ] Проверено, что изменения не приводят к утечке чувствительных данных в логи.
- [ ] Проверено, что изменения не приводят к утечке чувствительных данных в ошибки.
- [ ] Проверено, что изменения не приводят к утечке чувствительных данных в ответы API.
- [ ] Проверено, что изменения не приводят к утечке чувствительных данных в тестовые фикстуры.
- [ ] Проверено, что реализация соответствует актуальной ADR, архитектурным решениям и документации.
